## 3. 文档目的

本文档用于说明本系统的数据库模型、核心存储机制、数据流转规则与开发约束，供后续功能开发、数据库重构、接口联调与维护时参考。

本文档重点解决以下问题：

1. 系统有哪些核心数据对象  
2. 这些对象分别存在哪些表中  
3. 表之间如何关联  
4. 功能开发时数据应该如何落库  
5. 新功能开发时应遵守哪些数据库约束  
6. 如何避免旧结构继续扩散，导致数据库语义混乱

## 4. 系统存储设计目标

本系统数据库设计围绕“**用户—文件夹—周记录—工作碎片—周报历史**”的业务链路展开，目标如下：

### 4.1 清晰建模

将碎片记录本中的核心业务对象拆分为独立模型，不再混合存储。

### 4.2 用户隔离

所有业务数据均明确归属于用户，支持多用户使用与权限隔离。

### 4.3 原始数据与派生数据分层

工作碎片属于原始数据，周报属于派生结果，两者分别存储且可追溯。

### 4.4 便于扩展

后续可在现有结构上扩展：

- 月报生成
- 标签统计
- 进度分析
- 项目归档
- 附件系统
- 评论系统

### 4.5 支撑工程化维护

所有数据库变更采用迁移脚本管理，减少“手改数据库”带来的不可控风险。

## 6. 存储设计原则

### 6.1 数据对象单一语义

每张核心业务表只承载一种明确业务语义，不允许长期混合承担“文件夹 + 周记录”这类复合职责。

### 6.2 命名统一

数据库统一采用以下命名规则：

- 表名：复数、snake_case
- 字段名：snake_case
- 主键：`id`
- 外键：`xxx_id`
- 时间字段：`created_at` / `updated_at` / `deleted_at`

### 6.3 用户归属明确

所有用户业务数据表都必须包含 `user_id` 字段，并在代码层强制校验归属。

### 6.4 逻辑删除优先

默认采用逻辑删除，删除时设置 `deleted_at`，正常查询统一过滤已删除数据。

### 6.5 原始数据优先保留

所有派生数据必须可追溯到原始数据来源，派生结果不能替代原始数据。

### 6.6 变更版本化

数据库结构修改必须通过迁移脚本执行，不允许正式环境靠手工改表维持一致性。

## 9. 数据模型总体关系

### 9.1 逻辑关系图

```text
users (1)
 ├── memo_folders (N)
 │    └── memo_week_records (N)
 │         ├── memo_fragments (N)
 │         └── summary_records (0..N)
 │
 └── 用户设置字段（直接存于 users）
```

### 9.2 关系说明

- 一个用户可以拥有多个文件夹
- 一个文件夹可以包含多个周记录
- 一个周记录可以包含多条工作碎片
- 一个周记录可以生成一条或多条历史周报记录
- 一条历史周报可以追溯到一个来源周记录

## 11. 存储机制说明

### 11.1 碎片记录存储机制

#### 11.1.1 最小存储单元

系统以 `memo_fragments` 为最小业务存储单元。

#### 11.1.2 写入流程

```
用户选择某个周记录
→ 输入一条工作碎片
→ 前端提交 weekRecordId、workDate、title、content 等字段
→ 后端校验当前用户是否拥有该 weekRecord
→ 校验通过后写入 memo_fragments
```

#### 11.1.3 读取流程

```
用户点击某个周记录
→ 前端请求该周记录的全部碎片
→ 后端按 user_id + week_record_id + deleted_at is null 查询
→ 按 work_date、sort_order 排序后返回
```

------

### 11.2 周记录存储机制

#### 11.2.1 存储对象

```
memo_week_records
```

#### 11.2.2 存储内容

- 周记录标题
- 所属文件夹
- 周日期范围
- 当前周报内容
- 周记录状态

#### 11.2.3 设计意义

周记录是碎片的聚合层，不只是一个“列表容器”，还承担：

- 周级别状态管理
- 周报结果缓存
- 页面主视图绑定对象

------

### 11.3 周报生成存储机制

#### 11.3.1 原始数据来源

周报生成的数据来源是 `memo_fragments` 中的一周碎片，而不是手工输入的大段文本。

#### 11.3.2 生成流程

```
根据 weekRecordId 查询本周全部碎片
→ 按日期和排序组织输入内容
→ 调用 LLM 生成专业周报
→ 写回 memo_week_records.summary_content
→ 同步写入 summary_records（历史存档）
```

#### 11.3.3 双写策略

生成后的周报通常写入两个位置：

##### A. `memo_week_records.summary_content`

用途：

- 保存该周当前最新的周报正文
- 页面回显快
- 刷新页面后可直接查看

##### B. `summary_records`

用途：

- 历史记录存档
- 导出
- 查询
- 来源追溯

#### 11.3.4 设计原因

- 只写历史记录：当前周记录页不方便直接展示最近结果
- 只写当前周记录：缺乏历史归档能力

因此采用：

**当前缓存 + 正式存档** 的双层存储方式。

------

## 13. 数据约束规则

### 13.1 用户归属约束

所有业务表查询和更新必须校验 `user_id` 归属。

禁止：

- 前端传 `user_id` 决定权限
- 未带用户条件的更新与删除
- 跨用户直接访问资源

### 13.2 删除约束

默认逻辑删除，正常查询统一附加：

```
deleted_at is null
```

### 13.3 枚举值约束

以下字段必须在应用层进行枚举校验：

- `users.status`
- `memo_week_records.status`
- `memo_fragments.status`
- `memo_fragments.priority`

### 13.4 排序约束

凡涉及列表渲染的表，排序一律使用 `sort_order`。
 禁止以“创建时间自然顺序”作为正式排序逻辑。

### 13.5 时间字段约束

- `created_at`：创建时写入
- `updated_at`：更新时刷新
- `deleted_at`：逻辑删除时写入

## 14. 新功能开发数据库设计规则

这部分是后续开发最重要的参考。

### 14.1 不允许继续扩散旧语义

新功能中不得继续将以下旧概念作为正式主模型：

- `MemoGroup`
- `groupId`
- `folderName`
- `weekDate`

正式语义统一为：

- `folder`
- `weekRecord`
- `fragment`

### 14.2 新功能优先复用现有核心表

开发新功能时，优先判断是否能基于以下主表扩展：

- `memo_folders`
- `memo_week_records`
- `memo_fragments`
- `summary_records`

不要因为一个页面就临时新增含义模糊的新表。

### 14.3 原始数据与派生数据分离

例如：

- 工作碎片：原始数据
- 周报内容：派生数据
- 统计报表：派生数据

原则：

- 原始数据必须保留
- 派生数据必须标记来源
- 派生数据不得替代原始数据

### 14.4 新功能必须挂靠明确父对象

例如：

- 周记录附件 → 挂在 `memo_week_records`
- 碎片评论 → 挂在 `memo_fragments`
- 历史周报导出记录 → 挂在 `summary_records`

禁止设计“无归属主对象”的漂浮数据表。

### 14.5 接口语义必须与数据库一致

前后端交互统一优先使用以下字段：

- `folderId`
- `weekRecordId`
- `workDate`
- `sortOrder`

避免新旧字段混用。

## 15. 典型数据流

### 15.1 创建文件夹

```
前端提交文件夹名称
→ 后端校验用户身份
→ 写入 memo_folders
→ 返回新文件夹信息
```

### 15.2 创建周记录

```
前端提交 folderId、title、weekStartDate、weekEndDate
→ 后端校验 folder 归属
→ 写入 memo_week_records
→ 返回周记录详情
```

### 15.3 新增工作碎片

```
前端提交 weekRecordId、workDate、title、content
→ 后端校验 weekRecord 归属
→ 写入 memo_fragments
→ 返回碎片详情
```

### 15.4 生成周报

```
前端请求 generate-summary
→ 后端查询该周全部碎片
→ 拼接提示词
→ 调用 LLM
→ 写入 memo_week_records.summary_content
→ 同步写入 summary_records
→ 返回周报结果
```

### 15.5 查看历史周报

```
前端请求历史记录列表
→ 后端查询 summary_records
→ 返回历史结果
→ 如存在 source_week_record_id，可跳转来源周记录
```

## 16. 兼容旧结构的过渡说明

当前系统历史上存在：

- `memo_group`
- `memo_fragment`
- 以及围绕旧结构形成的部分代码

因此在迁移阶段允许存在兼容层，但必须遵循以下规则：

### 16.1 兼容层仅用于迁移

旧实体、旧接口、旧表只用于过渡，不能作为新功能主入口。

### 16.2 新文档统一采用新模型术语

后续所有数据库、接口、开发文档统一使用：

- `memo_folders`
- `memo_week_records`
- `memo_fragments`

### 16.3 迁移完成后应逐步废弃旧结构

包括：

- 废弃 `MemoGroup`
- 归档旧表
- 清理旧字段调用

## 18. 开发者检查清单

在新增任何数据库相关功能前，开发者必须先检查：

1. 该功能对应的是原始数据还是派生数据
2. 是否已有表可以承载该数据
3. 是否必须带 `user_id`
4. 是否需要逻辑删除
5. 是否需要排序字段
6. 是否存在明确父对象
7. 是否需要来源追溯
8. 命名是否符合 snake_case
9. 是否需要新增迁移脚本
10. 是否会继续扩散旧 `group` 语义

如果第 10 条答案是“会”，则说明设计方向已经偏了，需要先修正模型再开发。

## 19. 总结

本系统数据库存储采用“**原始碎片分层存储、周记录聚合管理、周报结果双层保存、用户维度强隔离**”的设计方案。

核心链路如下：

```
用户
→ 文件夹
→ 周记录
→ 工作碎片
→ 周报生成
→ 历史存档
```

该设计既支撑当前碎片记录本与专业周报生成功能，也为后续扩展以下能力提供稳定基础：

- 月报生成
- 标签统计
- 进度分析
- 项目归档
- 附件系统
- 评论系统
- 更多输出模板

后续新功能开发必须以本文档为数据库与存储设计基准，避免再次出现字段命名不一致、表语义混乱、旧结构反复扩散的问题。
