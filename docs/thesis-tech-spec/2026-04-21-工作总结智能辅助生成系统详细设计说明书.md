- 为后续维护者快速理解代码提供索引；

## 2. 后端详细设计

## 2.1 认证模块

### 2.1.1 控制器设计

类：`com.smartsummary.controller.AuthController`

接口：

1. `POST /api/auth/register`
2. `POST /api/auth/login`

请求体结构：

```json
{
  "username": "string",
  "password": "string",
  "email": "string"
}
```

响应结构：

```json
{
  "success": true,
  "data": {
    "id": 1,
    "username": "demo"
  }
}
```

### 2.1.2 服务设计

类：`com.smartsummary.service.UserService`

关键方法：

- `register(username, password, email)`
- `login(username, password)`
- `updateUserInfo(params)`
- `changePassword(userId, oldPassword, newPassword)`
- `getUserInfo(userId)`

### 2.1.3 处理逻辑

#### 注册逻辑

1. 查询用户名是否已存在；
2. 使用 `PasswordEncoder` 对密码加密；
3. 写入默认模型配置：
   - `modelId = qwen2.5-7b-instruct`
   - `baseUrl = http://localhost:8000`
   - `temperature = 0.7`
   - `maxTokens = 2048`
4. 持久化到 `users` 表。

#### 登录逻辑

1. 根据用户名查询用户；
2. 若不存在则抛出异常；
3. 使用 `passwordEncoder.matches` 验证密码；
4. 返回用户对象。

### 2.1.4 当前实现边界

- 登录后未生成令牌；
- 后端未建立标准会话；
- 前端依赖本地存储和请求头传递用户上下文。

## 2.2 当前用户解析模块

### 2.2.1 类说明

类：`com.smartsummary.service.CurrentUserService`

### 2.2.2 设计职责

用于在请求处理过程中解析当前用户身份，是便签相关接口的统一身份入口。

### 2.2.3 处理流程

1. 从 `RequestContextHolder` 读取当前请求；
2. 读取请求头 `X-User-Id`；
3. 解析为 `Long`；
4. 查询用户是否存在；
5. 返回合法的 `userId`；
6. 若任一步失败则抛出运行时异常。

### 2.2.4 设计意义

该类将“当前用户解析”从 Controller 中抽离出来，使便签接口不再依赖前端在请求体中显式传入用户 ID。

## 2.3 文件夹管理模块

### 2.3.1 控制器接口

类：`MemoController`

相关接口：

- `GET /api/memo/folders`
- `POST /api/memo/folders`
- `PUT /api/memo/folders/{id}`
- `DELETE /api/memo/folders/{id}`

### 2.3.2 服务类

类：`MemoFolderService`

关键方法：

- `listByUserId`
- `createFolder`
- `renameFolder`
- `updateCollapsed`
- `updateSortOrder`
- `deleteFolder`
- `getFolderDetail`

### 2.3.3 创建逻辑

1. 由 `CurrentUserService` 解析用户；
2. 若名称为空，默认使用 `Untitled Folder`；
3. 若排序号为空，读取当前用户最大排序号并加一；
4. 默认 `isCollapsed = 0`；
5. 写入 `memo_folders`。

### 2.3.4 删除逻辑

删除文件夹采用级联软删除：

1. 查询文件夹；
2. 找到其下所有未删除的周记录；
3. 对每条周记录下的未删除碎片设置 `deleted_at`；
4. 对周记录本身设置 `deleted_at`；
5. 对文件夹设置 `deleted_at`。

### 2.3.5 设计说明

该实现避免了物理删除造成的历史链路断裂，也使得数据迁移与问题排查更安全。

## 2.4 周记录管理模块

### 2.4.1 控制器接口

- `GET /api/memo/weeks?folderId=...`
- `GET /api/memo/weeks/{id}`
- `POST /api/memo/weeks`
- `PUT /api/memo/weeks/{id}`
- `DELETE /api/memo/weeks/{id}`
- `POST /api/memo/weeks/{id}/generate-summary`
- `POST /api/memo/weeks/{id}/save-summary`

### 2.4.2 服务类

类：`MemoWeekRecordService`

关键方法：

- `listByFolderId`
- `createWeekRecord`
- `getWeekRecordDetail`
- `updateWeekRecord`
- `deleteWeekRecord`
- `generateWeeklySummary`
- `saveSummary`

### 2.4.3 创建逻辑

若创建请求未携带周起止日期：

1. 系统读取当天日期；
2. 自动计算本周周一；
3. 自动计算本周周日；
4. 默认标题为 `start~end Weekly Notes`；
5. 初始状态为 `draft`；
6. `summaryContent` 置空；
7. `version` 初始化为 `0`。

### 2.4.4 查询逻辑

`getWeekRecordDetail` 会同时校验：

- `id` 是否匹配；
- `user_id` 是否匹配；
- `deleted_at IS NULL`。

### 2.4.5 更新逻辑

更新采用“取当前值 + 局部覆盖 + LambdaUpdateWrapper 回写”的模式，支持对以下字段进行部分更新：

- 所属文件夹；
- 标题；
- 周起止日期；
- 状态；
- 周报正文。

### 2.4.6 删除逻辑

删除周记录时：

1. 校验归属；
2. 先调用 `memoFragmentService.softDeleteByWeekRecord`；
3. 再将周记录本身标记为软删除。

### 2.5.1 控制器接口

- `GET /api/memo/fragments?weekRecordId=...`
- `POST /api/memo/fragments`
- `PUT /api/memo/fragments/{id}`
- `DELETE /api/memo/fragments/{id}`

### 2.5.2 服务类

类：`MemoFragmentService`

关键方法：

- `listByWeekRecordId`
- `createFragment`
- `updateFragment`
- `deleteFragment`
- `batchUpdateSort`
- `softDeleteByWeekRecord`

### 2.5.3 创建逻辑

默认值规则如下：

- `workDate` 默认当天；
- `title` 默认 `Untitled Fragment`；
- `content` 默认空串；
- `status` 默认 `done`；
- `priority` 默认 `medium`；
- `sortOrder` 默认取同周记录下最大排序号 + 1。

### 2.5.4 归属校验逻辑

在查询、创建、更新碎片前，都会通过 `assertWeekOwnership` 确认：

- 周记录存在；
- 周记录归属当前用户；
- 周记录未被删除。

### 2.5.5 排序设计

当前实现支持两种排序方式：

1. 服务层中的通用 `batchUpdateSort`；
2. 前端在同一天碎片拖动或上下移动后，通过多次 `PUT /api/memo/fragments/{id}` 更新排序号。

## 2.6 智能生成模块

## 2.6.1 通用总结生成

控制器：`SummaryController.generate`

接口：`POST /api/generate`

请求结构：

```json
{
  "text": "工作内容",
  "style": "list",
  "userId": 1
}
```

处理流程：

1. 若携带 `userId`，则从 `users` 表读取模型配置；
2. 调用 `llmService.normalizeStyle` 规范化风格；
3. 调用 `llmService.generateSummary`；
4. 若携带用户，则将输入文本与输出结果写入 `summary_records`；
5. 返回生成结果。

## 2.6.2 周报生成

控制器：`MemoController.generateSummary`

接口：`POST /api/memo/weeks/{id}/generate-summary`

请求结构：

```json
{
  "projectName": "Smart Summary",
  "userPosition": "后端开发"
}
```

处理流程：

1. 校验周记录归属；
2. 查询该周全部碎片；
3. 若碎片为空则抛出异常；
4. 构造周报输入文本；
5. 构建用户模型配置；
6. 调用 `llmService.generateWeeklySummary`；
7. 将结果保存回周记录；
8. 调用 `upsertSummaryRecord` 写入历史表；
9. 返回结果与统计信息。

### 2.6.3 周报输入构造规则

方法：`buildWeeklySummaryInput`

输入文本包含：

- 周起止日期；
- 周记录标题；
- 项目名称；
- 用户岗位；
- 每条碎片的日期、状态、标签、标题和内容。

构造后的文本示意：

```text
Week Range: 2026-04-06 to 2026-04-12
Week Record: 2026W14
Project: Smart Summary
Position: backend developer

Fragments:
- [2026-04-07][done][开发] 完成登录接口: 修复跨页跳转问题
```

### 2.6.4 历史写回规则

方法：`upsertSummaryRecord`

处理逻辑：

1. 读取当前周记录下全部碎片；
2. 按日期、排序号升序拼接原始文本；
3. 按 `user_id + source_week_record_id` 找最近一条记录；
4. 若不存在则新建；
5. 若存在则更新原文、总结文和模型配置快照。

### 2.6.5 模型调用封装

类：`LlmService`

关键方法：

- `generateSummary`
- `generateWeeklySummary`
- `normalizeStyle`
- `executeChat`
- `buildPrompt`
- `getChatCompletionsUrl`

### 2.6.6 请求体设计

请求体遵循 OpenAI 兼容风格：

```json
{
  "model": "qwen2.5-7b-instruct",
  "messages": [
    { "role": "system", "content": "..." },
    { "role": "user", "content": "..." }
  ],
  "temperature": 0.7,
  "max_tokens": 2048
}
```

### 2.6.7 风格处理

当前系统支持两种主风格：

- `list`
- `table`

其中钉钉、飞书、企微等别名最终也会归一到 `list`。

### 2.6.8 默认模型配置

若用户未设置完整模型参数，则 `UserConfig` 提供默认值：

- `baseUrl`: `http://localhost:8000`
- `modelId`: `qwen2.5-7b-instruct`
- `apiKey`: `lm-studio`
- `temperature`: `0.7`
- `maxTokens`: `2048`

## 4. 前端详细设计

## 4.1 请求层设计

文件：`src/services/http.js`

设计要点：

1. 创建统一 Axios 实例；
2. 30 秒超时；
3. 请求拦截器从 `localStorage.user` 取用户；
4. 自动注入 `X-User-Id`。

### 4.1.1 设计意义

这样做可避免每个业务请求重复拼接用户头，提高当前轻量登录方案下的接口一致性。

## 4.2 路由与鉴权设计

文件：`src/router/index.js`

关键点：

- `/app/*` 路由声明 `meta.requiresAuth = true`
- 全局前置守卫检查 `localStorage.user`
- 未登录跳转 `/login`

## 4.3 便签工作台设计

核心文件：

- `src/views/MemosView.vue`
- `src/views/memos/useMemoWorkspace.js`

### 4.3.1 结构设计

工作台主要由以下区域构成：

1. 左侧导航栏：文件夹与周记录；
2. 顶部上下文头：生成、保存、复制、导出、归档；
3. 中部筛选与时间线视图；
4. 底部快捷输入框；
5. 弹窗式创建/编辑对话框。

### 4.3.2 状态设计

`useMemoWorkspace` 维护了以下核心状态：

- `folders`
- `weeksByFolder`
- `selectedFolderId`
- `selectedWeekId`
- `currentWeek`
- `fragments`
- `summaryContent`
- `quickText`
- `generating`
- `activeDate`
- `filters`

### 4.3.3 数据加载流程

组件挂载后：

1. 读取本地用户；
2. 加载文件夹列表；
3. 逐个文件夹加载周记录；
4. 尝试优先选中“包含今天日期”的周记录；
5. 若没有，则尝试恢复本地缓存的选中周记录；
6. 选中后再加载详情与碎片。

### 4.3.4 交互设计亮点

1. 支持快捷输入工作碎片；
2. 支持按状态、标签、关键字筛选；
3. 支持跳转到当前日期；
4. 支持同日碎片上移、下移和重排；
5. 生成成功后自动将结果写入本地缓存，并跳转到生成页继续加工。

## 4.4 生成页设计

核心文件：`src/pages/generate/GeneratePage.vue`

### 4.4.1 页面职责

- 接收自由文本输入；
- 接收从便签工作台带来的周报源数据；
- 调用通用生成接口；
- 渲染 Markdown 结果；
- 提供复制、Markdown 导出、PDF 导出能力。

### 4.4.2 附件式上下文设计

从便签工作台触发生成后，系统会在 `localStorage.smart-summary:generate-result` 中写入：

- 来源标识；
- 周标题；
- 生成结果；
- 原始源文本；
- 碎片数量；
- 来源周记录 ID；
- 创建时间。

生成页加载时会读取该对象，并将其作为“已加载周记录素材”的附件显示。

### 4.4.3 请求拼接逻辑

若存在附件：

1. 先使用附件中的 `sourceText` 作为主输入；
2. 若用户又补充了文本，则附加为“补充要求”；
3. 最终合并后送往 `/api/generate`。

## 4.5 设置页设计

核心文件：`src/pages/settings/SettingsPage.vue`

### 4.5.1 页面结构

设置页分为三个标签页：

- 基本信息；

### 4.5.2 交互逻辑

- 页面加载时请求用户设置；

### 4.5.3 当前实现观察

前端 `settingsService` 中的方法定义与后端接口风格之间存在不完全一致：

- 前端 `save` 使用 `POST /api/settings`
- 后端实际为 `PUT /api/settings/info`
- 前端 `changePassword` 使用 `POST /api/settings/password`
- 后端实际为 `PUT /api/settings/password`

这说明设置模块前后端仍处于接口对齐演进阶段。后续如写论文，应表述为“当前版本仍在持续联调与优化”，不要写成完全统一成熟的接口体系。

## 5. 关键算法与规则设计

## 5.1 周默认日期计算

创建周记录时，若用户未指定周范围，则系统自动取当前日期所在周的周一至周日作为默认区间。

## 5.2 碎片排序规则

同一周记录下：

- 新建碎片默认排在末尾；
- 同一天的碎片允许手动调整顺序；
- 展示时按 `work_date ASC, sort_order ASC, created_at ASC` 排列。

## 5.3 统计规则

周记录统计信息基于碎片状态动态计算，包括：

- 总数；
- `todo` 数；
- `doing` 数；
- `done` 数；
- `blocked` 数。

## 5.4 提示词构造规则

通用总结生成与周报生成分别使用不同提示词模板，但共同遵守以下原则：

1. 不得编造未出现事实；
2. 优先保留结果、进展、问题和计划；
3. 语言应正式、清晰、可直接用于汇报；
4. 输出为正文，不附加解释。

## 7. 当前实现中的关键边界与风险

### 7.1 认证机制仍为轻量方案

当前系统依赖 `localStorage + X-User-Id`，适合作为开发期方案，但不适合作为生产级认证机制。

### 7.2 API Key 存储尚未加密

尽管接口回传时会做掩码处理，但数据库存储字段仍是明文 `api_key`。

### 7.3 设置接口仍需统一

设置页服务层和后端 REST 路径、HTTP 方法之间存在局部不一致。

### 7.4 生成过程为同步阻塞调用

模型调用会直接占用请求线程，若模型响应慢，用户等待时间会变长。

### 7.5 异常处理仍以运行时异常为主

当前控制器多以 `try-catch + message` 的方式返回错误，尚未形成统一异常中间层。

## 8. 可直接用于论文的技术结论

基于当前实现，可以稳定提炼出以下论文级技术事实：

1. 系统已形成完整的前后端分离架构；
2. 系统已完成从旧版 `memo_group / memo_fragment` 结构向新三层结构的迁移设计；
3. 系统已实现“原始碎片采集 -> 周记录聚合 -> 模型生成 -> 历史回溯”的完整闭环；
4. 系统通过 `source_week_record_id` 解决了历史结果可追溯问题；
5. 系统通过 `user_id` 和请求头用户解析实现了基础多用户隔离；
6. 系统具备一定工程化能力，但在认证、安全与接口统一性方面仍有明确后续优化空间。
