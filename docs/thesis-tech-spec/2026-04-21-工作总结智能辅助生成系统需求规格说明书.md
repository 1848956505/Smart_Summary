# 工作总结智能辅助生成系统需求规格说明书

## 1. 文档说明

### 1.1 文档目的

本文档用于从业务与功能角度定义“工作总结智能辅助生成系统”的建设目标、业务范围、核心角色、功能需求、非功能需求和约束条件。本文档面向以下使用场景：

- 作为论文撰写时的系统需求依据；
- 作为后续详细设计说明书的上游输入；
- 作为向 ChatGPT 或其他模型提供系统真实业务边界的基础材料；
- 作为前后端功能验证时的功能范围清单。

### 1.2 文档范围

本文档覆盖当前仓库中的实际系统范围，主要包含以下业务域：

1. 用户注册与登录；
2. 碎片记录本管理；
3. 周记录管理；
4. 工作碎片录入与维护；
5. 周报/工作总结智能生成；
6. 历史总结记录管理；
7. 个性化模型配置与连接测试。

本文档不覆盖以下内容：

- 第三方统一身份认证；
- 复杂权限角色体系；
- 多租户组织级管理；
- 附件上传与文件存储；
- 高并发分布式部署；
- 消息队列或异步任务调度。

### 1.3 编写依据

本文档基于以下真实资料整理：

- 项目代码仓库 `smart-summary` 与 `smart-summary-web`；
- 系统架构设计文档；
- 数据库存储设计书；
- 后端控制器、服务层、实体、迁移脚本、测试代码；
- 前端页面、组合式逻辑、服务层封装与路由逻辑。

## 2. 系统概述

### 2.1 建设背景

在实际工作和学习场景中，用户通常会产生大量零散的过程性记录，例如：

- 某天完成了某个接口联调；
- 修复了某个缺陷；
- 跟进了某项测试结果；
- 参加了会议并记录了后续行动项；
- 对某项需求进行了调研或文档整理。

这些记录通常具有以下问题：

1. 内容碎片化，分散于多个时间点；
2. 表述口语化，难以直接作为正式汇报材料；
3. 缺少统一结构，后期整理成本高；
4. 周报生成依赖人工总结，耗时且质量不稳定；
5. 已生成结果缺少与原始记录之间的追溯关系。

因此，需要构建一个能够实现“碎片记录沉淀 -> 周度整理 -> 智能生成 -> 历史回溯”的系统，以提高工作总结整理效率，并为后续论文撰写提供真实系统支撑。

### 2.2 系统目标

本系统的建设目标包括：

1. 提供多用户下的碎片化工作记录管理能力；
2. 提供文件夹、周记录、工作碎片三级结构化组织方式；
3. 提供基于大语言模型的周报/总结智能生成能力；
4. 将生成结果与原始工作记录保持可追溯关联；
5. 支持用户按自身模型环境进行个性化配置；
6. 通过分层架构和数据库迁移机制提升系统可维护性。

### 2.3 系统定位

本系统不是通用笔记应用，也不是纯粹的 AI 文本试玩页面，而是围绕“工作总结生成”这一业务场景构建的智能辅助系统。其核心定位如下：

- 前端负责承载碎片采集、周记录浏览、生成结果展示与历史回看；
- 后端负责业务编排、用户校验、数据持久化和模型调用封装；
- 数据库负责原始数据、派生数据和配置数据的存储；
- 大语言模型负责将结构化输入转换为规范化总结文本。

## 3. 业务范围与角色定义

### 3.1 业务主线

系统当前的核心业务主线为：

`用户 -> 文件夹 -> 周记录 -> 工作碎片 -> 周报生成 -> 历史记录`

对应关系如下：

- 用户：系统操作主体；
- 文件夹：一级分类容器；
- 周记录：按周聚合的业务容器；
- 工作碎片：最小原子化输入单元；
- 周报生成：由工作碎片构造模型输入并生成总结；
- 历史记录：保存生成结果及其来源信息。

### 3.2 用户角色

当前实现中仅存在单一业务角色“普通用户”，但不同用户之间的数据独立隔离。

用户具备以下权限：

- 注册与登录；
- 查看并修改自己的基础信息与模型配置；
- 创建、编辑、删除自己的文件夹、周记录与工作碎片；
- 生成自己的工作总结与周报；
- 查看和删除自己的历史记录。

### 3.3 数据隔离原则

系统采用“用户级数据归属”模式，不同用户的数据边界通过 `user_id` 控制。当前仓库中的实现方式主要表现为：

1. 业务表均带有 `user_id` 字段；
2. 后端接口中，碎片记录相关能力优先从请求头 `X-User-Id` 获取当前用户；
3. 服务层会对文件夹、周记录、碎片的归属关系进行校验；
4. 历史记录查询按 `user_id` 过滤。

## 4. 业务对象定义

### 4.1 用户

用户是系统登录主体，负责发起记录、生成和配置相关操作。

核心属性：

- 用户名；
- 密码；
- 邮箱；
- 头像；
- 岗位；
- 模型配置。

### 4.2 文件夹

文件夹是工作记录的一级分类容器，用于按主题、项目或任务域组织多个周记录。

核心属性：

- 名称；
- 排序号；
- 折叠状态；
- 用户归属。

### 4.3 周记录

周记录是系统中的核心聚合对象，用于承载某一周时间范围内的工作碎片和该周生成的总结内容。

核心属性：

- 所属文件夹；
- 标题；
- 周开始日期；
- 周结束日期；
- 当前状态；
- 周报正文；
- 版本号。

### 4.4 工作碎片

工作碎片是最小输入单元，用于记录单个工作事项或进展。

核心属性：

- 工作日期；
- 标题；
- 内容；
- 状态；
- 优先级；
- 标签；
- 排序号。

### 4.5 历史总结记录

历史总结记录用于保存生成后的结果，是从工作碎片派生出的结果数据。

核心属性：

- 原始输入文本；
- 输出总结文本；
- 风格；
- 模型配置快照；
- 来源周记录标识。

## 5. 功能需求

## 5.1 用户注册与登录

### 5.1.1 功能目标

系统应支持用户完成注册与登录，并为后续业务访问提供身份上下文。

### 5.1.2 功能要求

1. 系统应支持通过用户名、密码和邮箱完成注册。
2. 系统应校验用户名是否已存在。
3. 系统应在注册时对密码进行加密存储。
4. 系统应支持用户名与密码登录。
5. 登录成功后，前端应保存当前用户基础信息。
6. 未登录用户不应访问系统主工作区。

### 5.1.3 当前实现约束

当前实现采用轻量登录态桥接方案：

- 后端注册与登录接口位于 `/api/auth/register` 与 `/api/auth/login`；
- 前端将登录用户对象保存到 `localStorage.user`；
- 业务接口通过请求头 `X-User-Id` 传递当前用户标识；
- 尚未引入 JWT、Session 或 Refresh Token 机制。

## 5.2 基础信息与模型配置管理

### 5.2.1 功能目标

系统应允许用户维护个人资料与模型参数，从而适配不同模型服务环境。

### 5.2.2 功能要求

1. 系统应支持查看当前用户信息；
2. 系统应支持修改用户名、邮箱、岗位、头像等基础信息；
3. 系统应支持修改模型 ID、API Key、Base URL、Temperature、Max Tokens；
4. 系统应支持修改登录密码；
5. 系统应支持测试模型服务连通性；
6. 敏感信息不应在展示接口中以明文形式泄露。

### 5.2.3 当前实现特征

- 设置接口主要位于 `/api/settings`；
- 用户模型参数直接保存在 `users` 表中；
- 连通性测试通过调用 `{baseUrl}/v1/models` 进行；
- 当前实现对 API Key 的安全保护仍是轻量级的，数据库中存的是明文字段 `api_key`，不是加密字段。

## 5.3 文件夹管理

### 5.3.1 功能目标

系统应支持用户对工作记录进行一级分类管理。

### 5.3.2 功能要求

1. 系统应支持创建文件夹；
2. 系统应支持修改文件夹名称；
3. 系统应支持调整文件夹排序；
4. 系统应支持设置文件夹折叠/展开状态；
5. 系统应支持删除文件夹；
6. 删除文件夹时，应同步删除其下属周记录及工作碎片的有效性。

### 5.3.3 删除规则

当前实现采用软删除思路：

- 删除文件夹时，将其 `deleted_at` 标记为当前时间；
- 其下关联的周记录与工作碎片也同步标记 `deleted_at`；
- 查询时统一过滤 `deleted_at IS NULL` 的有效数据。

## 5.4 周记录管理

### 5.4.1 功能目标

系统应支持按周组织工作材料，并作为后续生成周报的核心容器。

### 5.4.2 功能要求

1. 系统应支持创建周记录；
2. 系统应支持维护标题、归属文件夹、周起止日期；
3. 系统应支持维护周记录状态；
4. 系统应支持保存周报正文；
5. 系统应支持删除周记录；
6. 删除周记录时，应同步删除其下工作碎片的有效性。

### 5.4.3 默认行为

当前实现中，若创建周记录时未显式传入日期，则系统自动以“本周周一到周日”作为默认范围。

## 5.5 工作碎片管理

### 5.5.1 功能目标

系统应支持用户按日录入工作碎片，并对其进行排序、筛选和维护。

### 5.5.2 功能要求

1. 系统应支持新增工作碎片；
2. 系统应支持编辑工作日期、标题、内容、状态、优先级、标签和排序号；
3. 系统应支持删除工作碎片；
4. 系统应支持按工作日期与排序号展示碎片；
5. 系统应支持快捷输入工作碎片；
6. 系统应支持同一天内的碎片重排；
7. 系统应支持按状态、标签和关键字筛选查看；
8. 系统应限制用户只能访问和操作自己的工作碎片。

### 5.5.3 状态范围

当前实现中主要使用以下状态：

- `todo`
- `doing`
- `done`
- `blocked`

### 5.5.4 优先级范围

当前实现中默认优先级为 `medium`，可扩展为多级优先级，但当前业务中未形成复杂规则引擎。

## 5.6 周报/工作总结智能生成

### 5.6.1 功能目标

系统应支持将原始工作碎片整理为结构化输入，调用大语言模型生成专业化周报或总结文本。

### 5.6.2 功能要求

1. 系统应支持普通文本输入生成总结；
2. 系统应支持基于某条周记录的碎片数据生成周报；
3. 系统应支持列表风格与表格风格两种输出偏好；
4. 系统应读取用户配置的模型参数；
5. 系统应在生成后保存生成结果；
6. 系统应支持生成结果复制、导出和二次编辑；
7. 系统应在无可用碎片时禁止周报生成。

### 5.6.3 当前实现链路

当前仓库中实际存在两条生成链路：

1. 通用总结生成链路：
   - 前端页面 `/app/generate`
   - 后端接口 `/api/generate`
   - 输入为任意文本 + 风格

2. 基于周记录的周报生成链路：
   - 前端便签工作台中触发
   - 后端接口 `/api/memo/weeks/{id}/generate-summary`
   - 输入来源于周记录与其下的工作碎片

### 5.6.4 生成结果处理

系统生成后的结果应支持：

- 回显到前端页面；
- 保存到周记录 `summary_content`；
- 写入 `summary_records` 历史表；
- 携带模型配置快照；
- 携带来源周记录标识。

## 5.7 历史记录管理

### 5.7.1 功能目标

系统应支持用户回看已生成内容，并为后续复用、复制、论文案例展示提供依据。

### 5.7.2 功能要求

1. 系统应支持查看历史记录列表；
2. 系统应支持查看单条历史详情；
3. 系统应支持删除历史记录；
4. 系统应优先展示最近生成记录；
5. 系统应保留生成时的模型参数快照；
6. 系统应尽可能保留与来源周记录的关联。

### 5.7.3 当前实现限制

当前历史接口中：

- 列表接口按用户查询，最多返回最近 50 条；
- 详情接口按记录 ID 读取；
- 删除接口按记录 ID 执行逻辑删除；
- 列表与详情会将返回对象中的 `apiKey` 处理为 `***`。

## 6. 非功能需求

## 6.1 易用性

1. 页面结构应支持快速进入主业务路径；
2. 周记录与碎片层级结构应清晰；
3. 常用操作应尽量在同一工作台完成；
4. 生成结果应具备清晰展示区域；
5. 支持快捷录入以降低记录成本。

## 6.2 响应性能

1. 常规页面查询操作应在可接受时间内完成；
2. 模型调用过程应提供加载状态；
3. 模型连接测试应返回连接状态与延迟信息；
4. 前端应具备基本的错误提示机制。

## 6.3 稳定性

1. 删除操作应优先采用软删除；
2. 模型调用失败不应导致原始记录丢失；
3. 周记录删除应保证碎片级联删除的一致性；
4. 数据迁移应支持幂等执行和回滚审视。

## 6.4 可维护性

1. 后端应保持 Controller / Service / Repository 分层；
2. 数据结构演进应通过 Flyway 管理；
3. 前端应按页面、组件、服务分层组织；
4. 核心模块应具备基础回归测试。

## 6.5 可扩展性

系统后续可扩展方向包括：

- 月报生成；
- 标签统计；
- 项目维度归档；
- 附件与文档引用；
- 更强的权限与登录机制；
- 异步生成与任务队列。

## 6.6 安全性

1. 用户不应访问其他用户的文件夹、周记录和碎片；
2. 登录密码必须加密存储；
3. 模型 API Key 不应在前端展示接口中裸露；
4. 后续应逐步从 `X-User-Id` 过渡到正式认证机制。

## 7. 约束条件

### 7.1 技术栈约束

当前系统实现基于以下栈：

- 后端：Java 17、Spring Boot 3.2.0、MyBatis-Plus、MySQL、Flyway、Spring Security、OkHttp
- 前端：Vue 3、Vue Router、Vite、Element Plus、Axios、html2pdf.js、marked

### 7.2 模型接口约束

当前 LLM 调用默认按 OpenAI 兼容接口风格组织请求：

- 模型列表探测：`/v1/models`
- 文本生成：`/v1/chat/completions`

### 7.3 部署约束

当前默认开发配置中：

- 后端运行端口为 `8080`
- MySQL 数据库名为 `smart_summary`
- 默认模型服务地址为 `http://localhost:8000`

## 8. 验收边界

若系统满足以下条件，则可认为核心需求已被实现：

1. 用户可成功注册、登录并进入系统主界面；
2. 用户可维护自己的文件夹、周记录和工作碎片；
3. 用户可基于工作碎片生成周报；
4. 生成结果可被保存和回看；
5. 不同用户的数据不会互相串读；
6. 模型参数可配置且可完成基本连通性测试。

## 9. 当前版本已知边界

为避免后续基于文档撰写论文时出现失真，需明确当前版本的真实边界：

1. 当前登录态是轻量方案，不是完整的令牌认证系统；
2. 用户设置与模型配置接口存在“前端调用风格”与“后端 REST 设计”尚未完全统一的迹象；
3. API Key 当前存储方式仍偏开发期实现，不属于完善的密钥托管方案；
4. 生成链路为同步调用，尚未引入异步任务队列或结果轮询；
5. 当前测试以回归测试和服务归属测试为主，尚非完整自动化测试体系。

以上边界在论文中应如实表述为“当前实现范围”或“后续可优化方向”，不宜夸大为已完成能力。
