我看完了，结论是：**你的微调模型确实有问题，但不是“差到 3.35”那种问题；分数偏低主要由“过度模板化 + 评审口径不匹配 + 忠实性严格扣分”共同造成。**

## 1. 评分结果本身暴露了两个主要扣分点

`pred_ft_summary.json` 显示，微调模型的：

## 2. 我抽查推理结果后，发现微调模型有明显“模板上头”问题

比如很多输出都会固定生成：

```text
# 📊 xxx周报
状态标签：🔄 进行中 | ⭕ 阻塞
优先级标签：🔴 P0 | 🟠 P1 | 🟡 P2

## 本周工作概览
## 本周工作明细
## 风险与待办问题
## 下一步行动
## 说明与建议
```

这个结构其实很清楚，所以 `structure` 高是合理的。

但问题是：**输入里没有状态标签、优先级标签、明确的下周完成时间、两周内目标、已启动流程等内容，模型经常自己补出来。**

比如原始输入只是说：

> 得赶紧找客户补救一下

模型容易写成：

> 已启动客户关系挽回机制，制定补救策略。

原始输入只是说：

> 得排查 JVM 优化问题

模型容易写成：

> 计划两周内完成初步排查并提出解决方案。

这些内容读起来很像正式周报，但对忠实性评分来说就是“无依据扩写”。所以 `faithfulness` 被压低并不奇怪。

## 3. `format` 被低估，主要是评审把“列表风格”理解成了“纯列表”

你的评估 prompt 里对格式的定义是：

> 要求列表时，是否正确输出列表结构；Markdown 格式是否规范；全文格式是否统一。

但它没有明确说明：

> 列表风格可以是“标题 + 小节 + 项目符号列表”的 Markdown 周报。

所以复核模型经常认为：
**“你要求列表风格，但模型输出了完整周报，所以格式不符合。”**

这其实是评审口径问题。因为你的任务本来就是“周报生成”，列表风格不应该等于“不要标题、不要小节、只输出几条短列表”。这个地方需要改评估 prompt。

格式要求

没有使用参考答案。

但如果后面你要跑 ROUGE、BLEU、BERTScore，这会直接出问题。也说明测试集生成脚本可能没有把 `output` 字段保留下来。这个要修，不然自动指标会变成“没有标准答案的小猫考试”，它喵半天也没法判卷。

# 现在该怎么改

## A. 先改推理 prompt，压住过度扩写

你现在的微调模型太喜欢“高级周报模板”。建议推理时给它加硬约束：

```text
请将下面的工作流水整理为列表风格周报。

要求：
1. 使用 Markdown 标题和项目符号列表组织内容；
2. 可以包含“本周工作概览、工作明细、风险与待办、下周计划”等小节；
3. 不要使用 emoji、状态标签、优先级标签和复选框；
4. 不要添加原始记录中没有的具体时间、负责人、完成期限、处理结果；
5. 对于尚未完成或不确定的事项，只能写“待排查”“待确认”“需继续跟进”；
6. 不要把“计划做”写成“已完成”或“已启动”；
7. 输出应简洁，不要过度扩写。
```

尤其是第 3、4、5、6 条，能明显提高忠实性。

------

## B. 改评估 prompt，让“列表风格”定义更清楚

把 format 部分改成这样：

```text
3. 格式符合度（format）
定义：考察输出是否符合指定的表格风格或列表风格要求。

判分标准：
- 若要求为“列表风格”，允许使用 Markdown 标题、小节标题、项目符号列表、编号列表组织内容；
- 列表风格不要求全文只能是纯项目符号，只要主体内容以列表方式组织，且没有使用 Markdown 表格，即可认为符合；
- 若要求为“表格风格”，应包含 Markdown 表格或清晰的表格化字段结构；
- 不应因为输出是完整周报而降低列表风格评分；
- 只有在格式混乱、表格/列表风格明显不符、Markdown 结构错误时才降低格式分。
```

这能解决“完整周报被当成不符合列表”的问题。

------

## C. 改 faithfulness 评分标准，区分“合理归纳”和“事实编造”

现在评审对忠实性比较狠，容易把正常归纳也当扩写。建议改成：

```text
5. 忠实性（faithfulness）
定义：考察输出内容是否忠于原始记录，是否存在无依据事实补充。

判分标准：
- 允许对原始记录进行职业化改写、归纳总结和结构化表达；
- 允许将口语化内容整理为正式表达；
- 不应因合理概括、同义改写、结构化归类而扣分；
- 只有当模型添加了原始记录中没有的具体事实时才扣分，例如：
  - 新增具体时间、负责人、完成期限；
  - 将待办写成已完成；
  - 添加未提及的处理结果、业务影响、量化指标；
  - 编造已启动、已完成、已上线、已验证等状态。
```

这样会更公平。

------

## D. 修复测试集生成：保留 reference 和 style

你的测试集最好每条都长这样：

```json
{
  "sample_id": 1,
  "instruction": "请将下面的工作流水转化为专业的周报。（当前周报风格：列表风格）",
  "input": "...",
  "output": "...标准答案...",
  "meta": {
    "style": "列表风格"
  }
}
```

推理结果里也必须保留：

```json
{
  "reference": "...标准答案...",
  "style": "列表风格"
}
```

否则后面传统自动指标没法跑。

------

## E. 评分脚本要处理评审失败样本

你的 `pred_ft_scored.jsonl` 有 200 条，但 summary 只有 198 条有效结果。原因是有两条评审 JSON 解析失败，被跳过了。summary 里也显示最终统计 count 是 198。

建议评审失败时自动重试一次，不要直接漏掉：

```python
for retry in range(2):
    try:
        primary_result = primary_client.judge_one(...)
        review_result = review_client.review_one(...)
        break
    except Exception as e:
        if retry == 1:
            scored = {
                **item,
                "resolved_style": style,
                "judge_error": str(e)
            }
```

或者给 judge API 加 `max_tokens=512`，避免 JSON 被截断。

# 最核心结论

你的微调模型当前不是“质量很差”，而是：

> **结构和专业表达明显学到了，但生成模板过重，容易添加状态、优先级、计划期限和处理动作等输入中没有的信息；同时评审 prompt 对“列表风格”的定义不够准确，导致格式分被压低。**

建议下一轮先做两件事：

1. **推理 prompt 加“禁止无依据扩写、禁止状态标签/优先级标签/emoji”**；
2. **评估 prompt 明确“列表风格可以是 Markdown 周报小节 + 项目符号列表”。**

改完以后，微调模型的 `format` 应该会上去；`faithfulness` 是否提升，主要取决于你能不能压住它“自作聪明补细节”的毛病。这个毛病不致命，但必须拴绳，小模型一正式起来就容易西装革履地胡说八道。
