# 当前数据集的定位与性质：

我建议最终确定为 **10 个特点**。

## 1. 内容忠实性

输出必须基于原始记录，不能编造、夸大或补充无依据事实。

禁止新增：

- 未出现的结果；
- 未出现的时间；
- 未出现的负责人；
- 未出现的量化指标；
- 未出现的业务价值；
- 未出现的完成状态。

这是数据集的第一底线。

------

## 2. 信息完整性

输出应尽量覆盖原始记录中的关键事项，包括：

- 已完成任务；
- 进行中任务；
- 遇到的问题；
- 风险和阻塞；
- 沟通事项；
- 后续计划。

不能只写积极内容，也不能自动过滤“看起来不好看”的问题。

------

## 3. 状态边界准确性

任务状态必须严格对应原文。

建议把状态分成几类：

| 原始状态                 | 输出应保持为          |
| ------------------------ | --------------------- |
| 已完成、搞定、修好了     | 已完成                |
| 正在做、还在看、持续排查 | 进行中                |
| 准备做、打算做、计划做   | 计划推进              |
| 待确认、没定、等反馈     | 待确认                |
| 卡住、报错、阻塞         | 存在问题 / 待解决     |
| 可能、大概、好像         | 不确定 / 需进一步确认 |

这条是防止“无依据扩写”的核心。

------

## 4. 不确定性保留

原文不确定，输出也不能写确定。

原文模糊，输出可以更清楚，但不能更肯定。

例如：

原文：

> 可能是接口参数的问题，还得再看看。

正确：

> 接口异常可能与参数配置有关，需继续排查确认。

错误：

> 已定位接口异常由参数配置导致。

------

## 5. 结构化重组能力

输出不是简单润色，而是要把碎片信息整理成清晰结构。

可以按：

- 工作模块；
- 项目；
- 时间；
- 任务状态；
- 问题与计划。

进行归类。

这体现你的任务价值：不是“改写一句话”，而是“整理一周工作”。

------

## 6. 格式稳定性

数据集应稳定支持两种基础格式：

### 列表风格

适合普通周报：

```markdown
## 本周工作
- ...

## 问题与风险
- ...

## 下周计划
- ...
```

### 表格风格

适合在线表格或结构化汇报：

```markdown
| 工作模块 | 工作内容 | 当前状态 | 后续计划 |
|---|---|---|---|
| ... | ... | ... | ... |
```

不要让同一样本中格式混乱，也不要让“表格风格”和“列表风格”边界不清。

------

## 7. 语言规范性

输出应将口语化表达转成正式、清晰、凝练的书面表达。

但要控制上限：

> 专业，不等于浮夸；正式，不等于包装；清晰，不等于脑补。

可以使用：

- 完成；
- 推进；
- 沟通；
- 排查；
- 整理；
- 跟进；
- 待确认；
- 持续优化。

谨慎使用或禁止无依据使用：

- 显著提升；
- 全面优化；
- 圆满完成；
- 重大突破；
- 深度赋能；
- 形成机制；
- 有效保障；
- 取得阶段性成果。

除非原始记录明确支撑。

------

## 8. 输入真实性与噪声多样性

输入不能太干净，要保留真实流水特征：

- 口语化；
- 重复；
- 顺序混乱；
- 信息不完整；
- 有临时事项；
- 有情绪表达；
- 有不确定信息；
- 有待办和阻塞；
- 有无关或低价值信息。

否则模型只能处理“考试题”，不能处理真实用户输入。

------

## 9. 岗位与场景覆盖

数据集要覆盖不同岗位和任务类型。

至少可以包括：

- 技术开发；
- 测试；
- 产品；
- 运营；
- 销售；
- 行政；
- 项目管理；
- 客服支持。

每类岗位的工作流水差异很大，不能让模型只学会“技术周报味”。

------

## 10. 风格边界清晰

基础数据集只训练：

> 忠实、完整、清晰、规范的周报初稿。

不训练：

- 报喜不报忧；
- 高情商话术；
- 领导汇报包装；
- 负面弱化；
- 成果拔高；
- 价值升华；
- 企业黑话。

这些应该放到系统层作为可选功能。

------

# 五、最终优先级

## P0：底线能力

必须先保证：

1. **内容忠实性**
2. **状态边界准确性**
3. **不确定性保留**
4. **信息完整性**

这四个决定模型是否可信。

## P1：可用能力

接着保证：

1. **结构化重组能力**
2. **格式稳定性**
3. **语言规范性**

这三个决定模型是否好用。

## P2：泛化与工程能力

后续完善：

1. **输入噪声多样性**
2. **岗位与场景覆盖**
3. **可评估字段完备性**

这些决定模型能不能推广到更多真实场景。

------

# 六、哪些内容交给系统层做

下面这些不适合写进基础训练数据：

| 内容             | 应该怎么做                         |
| ---------------- | ---------------------------------- |
| 给老板看的积极版 | 系统提供“积极汇报版”选项           |
| 弱化问题         | 系统提供“问题简写/完整呈现”选项    |
| 成果突出         | 系统提供“成果强化”二次润色         |
| 高情商表达       | 系统提供“委婉表达”风格             |
| 价值升华         | 系统提供“补充价值建议”，由用户确认 |
| 固定公司模板     | 系统提供模板选择                   |
| 领导偏好语气     | 系统提供个性化配置                 |

基础模型负责“说真话”。
系统功能负责“怎么说更合适”。
这俩别混，小猫戴两顶帽子会晕。

# 最终一句话版本

我建议最终定稿为：

> **该数据集应以“零幻觉的结构化转述”为核心目标，将真实、口语化、碎片化的工作流水，在不编造、不夸大、不改变任务状态的前提下，整理为结构清晰、格式稳定、语言规范的中性周报初稿。**

更简短一点：

> **输入真实多样，输出忠实完整，状态边界准确，结构格式稳定，语言专业但不替用户美化事实。**
