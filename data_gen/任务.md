# 开发方案：

## 一、重构目标

请将当前数据合成管线重构为：

**任务种子库 → 结构化种子标准化 → 控制画像采样 → 语义底稿构建 → input 渲染 → table/list 双 output 渲染 → 校验 → 导出**

必须满足以下要求：

### 目标 1：同一个 input 生成两种 output

每个样本必须生成两条训练数据：

- instruction: 请将以下工作记录整理为表格形式周报
- input: 原始流水账
- output: 表格形式周报

以及：

- instruction: 请将以下工作记录整理为列表形式周报
- input: 与上面完全相同
- output: 列表形式周报

该要求来自任务文档，必须实现。

### 目标 2：input 控制字段显式概率化

不要再只靠 prompt 里的“低概率出现”“适量加入”之类模糊描述。
必须把以下字段做成程序侧显式采样的 `control_profile`：

- 姓名是否出现
- 岗位名称是否显式出现
- 时间周期是否明确
- 日期标注情况（无/部分/全部）
- 指标类型（明确量化/模糊/无）
- 是否有明确风险项
- 风险应对中是否有时间点
- 风险应对中是否有负责团队
- 是否有明确计划项
- 计划期限类型（明确/模糊/无）
- 是否有协同事项
- 协同方是否明确
- 输入格式（散装/列表/段落）
- 信息完整度（高/中/低）
- 错别字噪声
- 情绪化噪声

这些分布必须与任务文档对齐。

### 目标 3：去除大段硬编码内容池

当前类似：

- `RISK_TITLES`
- `PLAN_TITLES`
- `METRIC_EXPLICIT_POOL`
- `METRIC_VAGUE_POOL`

这类内容不应再作为主生成来源。
主内容应来自：

- 结构化种子
- 语义底稿
- 模型按约束生成

允许保留极少量轻量池作为兜底，例如：

- `NAME_POOL`
- `TEAM_POOL`
- `TIMEPOINT_POOL`
- `DEADLINE_POOL`

当模型“没按要求生成”时，补位用。但仅用于补位，不作为主要语义来源。

## 二、当前系统存在的问题

Codex 在重构时，需要先理解当前问题，不要继续在旧架构上打补丁。

### 1. 当前 `pipline.py` 仍是按 style 分样本生成

现有流程是：

- 选风格
- 拼一个总 prompt
- 一次性生成 input + output

这种结构天然不适合“同一 input 生成双 output”。

### 2. 当前 input 控制逻辑不是真正可控

配置文件里虽然已经有：

- `NAME_MENTION_PROB`
- `TYPO_NOISE_PROB`
- `EMOTION_NOISE_PROB`
- `INCOMPLETE_INFO_PROB`
- `ROUTINE_TASK_PROB`

但这些还没有形成真正的结构化控制链。
特别是 `pipline.py` 中 prompt 格式化部分还存在明显错误，说明当前控制逻辑并未闭环。

### 3. 当前 `task_seeds.json` 是素材库，不是控制底稿

现有 `task_seeds.json` 质量可用，保留它，不要推翻重做。
但必须新增“结构化标准化层”，因为现在的 seed 只是自然语言事件句，不足以直接支撑字段级控制。

## 三、重构原则

### 原则 1：保留现有 seeds，不推翻

保留 `task_seeds.json` 作为原始语义素材库。
新增标准化步骤，将其转成结构化种子。

### 原则 2：程序决定结构，模型负责自然表达

不要让模型决定“这条样本有没有风险、有没有计划、有没有姓名”。
这些必须由程序先采样。

### 原则 3：input 与 output 解耦，但共用统一事实源

input 不是 output 的直接来源。
input 和两个 output 都应来源于同一个 `semantic_frame`。

### 原则 4：先生成 sample，再生成双风格

数据主键从“style”改为“sample_id”。

## 模块 3：`semantic_builder.py`

### 功能

### 验收标准

- `semantic_frame` 可序列化为 JSON

- 字段状态与 control_profile 匹配

```python
        export_train_record(sample_id, "list", input_text, list_output)
        export_debug_record(sample_id, input_text, table_output, list_output, semantic_frame, control)
```

#### 3. `list_output_prompt.txt`

## 十一、Codex 开发时的硬性要求

### 1. 不要继续沿用旧 `STYLE_MAP` 为主轴

旧代码是按风格跑全量样本，这次必须按 `sample_id` 跑。

### 2. 不要继续把所有配置放一个文件

拆文件。

### 3. 不要把风险/计划/指标主内容写死成固定池

可保留极小兜底池，但不能靠标题池生成核心语义。

### 4. 必须保留 debug 信息

否则后续你没法分析“这条样本为什么失败”。

### 5. 必须可复现

采样器、模型调用、导出都要保留随机种子信息。

## 十二、验收标准

项目完成后，至少满足以下标准：

### 数据层

- 每个 `sample_id` 生成两条训练样本
- 两条样本 input 完全一致
- output 分别是 table/list 风格

### 控制层

- 采样 1000 条后，核心字段分布与目标概率接近
- 例如：
  - 姓名出现率接近 15%
  - 风险出现率接近 60%
  - 计划出现率接近 70%
  - 协同出现率接近 40%

### 质量层

- input 明显不是正式周报
- table/list 风格明显不同
- 双 output 共享事实源，无明显互相矛盾

- 支持断点续跑
